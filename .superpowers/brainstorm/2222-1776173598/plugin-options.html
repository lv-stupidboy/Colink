完全插件化方案对比

新增Agent只写插件包，不改动Colink代码

方案对比

A

编译时链接 + Build Tags

插件包独立目录，通过Go build tags选择性编译 

// 插件包结构（独立目录，不在Colink仓库内）
internal-agent/
├── plugin.go       // +build internal_agent
├── adapter.go
└── config.go

// Colink 编译时
go build -tags "internal_agent" ./cmd/server

// 或默认编译（不含内网插件）
go build ./cmd/server

优点

  • Go原生支持，无额外依赖
  • 类型安全，IDE友好
  • 跨平台一致

缺点

  • 需要修改编译命令
  • 插件包需放在特定目录
B

独立进程 + RPC通信

插件作为独立进程，通过gRPC/HTTP与Colink通信 

// 插件独立进程
internal-agent-plugin/
├── main.go         // 启动RPC服务
├── adapter.go      // 实现AgentAdapter
└── proto/          // 定义接口协议

// Colink 配置
plugins:
  - type: internal_agent
    endpoint: localhost:50051  // RPC地址

// Colink 运行时连接插件进程
adapterRegistry.Connect("internal_agent", "localhost:50051")

优点

  • 完全独立，无需编译Colink
  • 插件可独立开发/部署
  • 支持不同语言实现

缺点

  • 进程管理复杂
  • 通信开销
  • 调试困难
C

配置驱动 + 外部Adapter进程

类似方案B，但Adapter进程由Colink启动管理 

// Colink配置文件
base_agents:
  - type: internal_agent
    name: 内网Agent
    cli_path: /path/to/internal-agent-adapter
    protocol: acp   # 或 custom

// Colink自动启动外部Adapter进程
// 通过ACP协议或自定义协议通信

优点

  • 配置即用，无代码改动
  • 进程由Colink管理
  • 支持ACP协议扩展

缺点

  • Adapter需实现完整进程协议
  • 需约定通信协议

推荐方案详解：方案C（配置驱动）

内网部署流程（零代码改动）

1. 开发内网Adapter（独立项目）
   internal-agent-adapter/
     main.go  → 读取stdin，输出ACP格式到stdout

2. 编译内网Adapter
   go build -o internal-agent-adapter.exe ./main.go

3. 部署到Colink同目录
   /opt/colink/
     ├── colink-server.exe
     ├── internal-agent-adapter.exe  ←── 新增
     └── data/configs/config.yaml

4. 配置Colink（无需重启服务）
   config.yaml:
     base_agents:
       - type: internal_agent
         name: 内网Agent
         cli_path: ./internal-agent-adapter.exe
         protocol: acp

5. 完成！Colink自动识别并使用
