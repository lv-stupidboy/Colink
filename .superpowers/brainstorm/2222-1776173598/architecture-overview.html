当前问题 vs 目标架构

当前架构问题

现状：硬编码工厂模式
┌─────────────────────────────────────────────────────────────┐
│  编排层 (Orchestrator / ExecutionService)                   │
│    ↓ 调用 NewAdapter(baseAgent)                             │
├─────────────────────────────────────────────────────────────┤
│  适配器工厂 (adapter.go)                                     │
│    switch baseAgent.Type {                                   │
│      case "claude_code":    → ClaudeAdapter     ❌ 硬编码   │
│      case "open_code":      → OpenCodeAdapter   ❌ 硬编码   │
│      case "open_code_acp":  → OpenCodeACPAdapter ❌ 硬编码  │
│      default: nil                                          │
│    }                                                        │
├─────────────────────────────────────────────────────────────┤
│  具体适配器实现                                              │
│    ClaudeAdapter      ←── 内部代码                          │
│    OpenCodeAdapter    ←── 内部代码 (要删除)                  │
│    OpenCodeACPAdapter ←── 内部代码 (要重命名)                │
└─────────────────────────────────────────────────────────────┘

问题清单

  • 违反开闭原则：新增Agent需修改工厂switch
  • 4+文件耦合：adapter.go、base_agent_service.go、configgen、model
  • 无法插件化：内网Agent无法独立打包部署
  • 类型混乱：OpenCode vs OpenCode(ACP) 对用户暴露
  • 命名不一致："Claude Code" 有空格

目标架构：插件化注册机制

目标：Adapter Registry + 插件包
┌─────────────────────────────────────────────────────────────┐
│  编排层 (Orchestrator / ExecutionService)                   │
│    ↓ adapterRegistry.GetAdapter(baseAgent.Type)             │
│    ✅ 完全不感知具体Agent类型                                 │
├─────────────────────────────────────────────────────────────┤
│  适配器注册中心 (AdapterRegistry)                            │
│    map[string]AdapterFactory                                │
│    Register(type, factory)  ←── 插件注册入口                 │
│    GetAdapter(type)         ←── 编排层调用                   │
├─────────────────────────────────────────────────────────────┤
│  插件包 (独立部署单元)                                        │
│                                                              │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐       │
│  │ claude-code  │  │ open-code    │  │ 内网Agent    │       │
│  │   插件包     │  │   插件包     │  │   插件包     │       │
│  │              │  │              │  │              │       │
│  │ Register()   │  │ Register()   │  │ Register()   │       │
│  │ ClaudeAdapter│  │ OpenCodeACP  │  │ 内网Adapter  │       │
│  │ config目录   │  │ config目录   │  │ config目录   │       │
│  └──────────────┘  └──────────────┘  └──────────────┘       │
│                                                              │
│  新增Agent：只需实现插件包 + 调用Register()                   │
└─────────────────────────────────────────────────────────────┘

架构优势

  • 开闭原则：新增Agent无需修改编排层代码
  • 插件独立：内网Agent可单独开发、打包、部署
  • 注册机制：启动时加载插件，动态注册类型
  • 统一接口：所有Adapter实现相同AgentAdapter接口
  • 配置分离：每个插件自带默认配置

ACP协议通用性论证

协议 输出格式 扩展性 适用场景
CLI原生 各Agent自定义格式
Claude: stream-json
OpenCode: 自定义JSON		 ❌ 每个Agent需单独解析 单一Agent
ACP协议 标准化JSON-RPC
统一事件类型
结构化输出		 ✅ 新Agent只需实现ACP 多Agent通用

结论：ACP（Agent Communication Protocol）提供标准化的事件流格式，包括：
• 工具调用/结果、思考过程、用量统计、错误信息
• 新Agent只需实现ACP协议，无需单独适配解析逻辑
