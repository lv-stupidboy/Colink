# 市场团队包导入冲突处理优化设计文档

**创建日期**: 2026-04-22
**状态**: 待审查

## 背景

当前市场管理下的团队包导入能力存在以下问题：
1. **单包导入**：预览只显示包内容信息，不检测本地冲突
2. **批量导入**：直接执行导入，无冲突处理选项

用户要求参考管理工具下的团队包导入能力，增加冲突检测和策略选择功能。

## 目标

- 单包导入：预览时显示冲突信息，用户可选择"覆盖"或"跳过"
- 批量导入：统一策略模式，所有包使用相同策略处理冲突

## 设计方案

### 方案选择

采用**方案A：后端增强 + 前端复用**，理由：
- 与管理工具保持一致的交互体验
- 后端统一处理冲突检测逻辑
- 改动范围可控，最大化复用现有代码

### 改动范围

| 模块 | 文件 | 改动内容 |
|------|------|----------|
| 后端 | `internal/service/teampackagesync/types.go` | 增加冲突检测字段 |
| 后端 | `internal/service/teampackagesync/service.go` | PreviewPackage 方法增加冲突检测逻辑 |
| 前端 | `web/src/types/index.ts` | PackagePreviewResponse 类型更新 |
| 前端 | `web/src/pages/Market/TeamPackages.tsx` | 预览弹框增加冲突展示和策略选择 |

type PreviewPackageResponse struct {

#### 1.2 Service 方法改造 (`service.go`)

在 `PreviewPackage` 方法中，解析完 manifest 后，检查本地资源是否存在：

```go
// 检查 Workflow 是否存在
workflows, _ := s.workflowRepo.FindAll(ctx)
for _, wf := range workflows {
    if wf.Name == manifest.Workflow.Name {
        response.Workflow.Exists = true
        break
    }
}

// 检查 Roles 是否存在（按名称匹配）
for i, role := range manifest.Roles {
    existing, _ := s.agentRepo.FindByName(ctx, role.Name)
    if existing != nil {
        response.Roles[i].Exists = true
        response.Roles[i].LocalId = existing.ID.String()
    }
}

// 检查各类 Assets 是否存在（按名称匹配）
for i, skill := range manifest.Assets.Skills {
    existing, _ := s.skillRepo.FindByName(ctx, skill.Name)
    if existing != nil {
        response.Assets.Skills[i].Exists = true
    }
}
// ... Commands, Subagents, Rules, Settings 同理
```

计算 `ConflictCount`：
```go
response.ConflictCount = 0
if response.Workflow.Exists {
    response.ConflictCount++
}
for _, role := range response.Roles {
    if role.Exists {
        response.ConflictCount++
    }
}
for _, skill := range response.Assets.Skills {
    if skill.Exists {
        response.ConflictCount++
    }
}
// ... 其他资产类型
```

**注意**：需要注入必要的 repo 到 SyncService（workflowRepo、agentRepo、skillRepo 等），目前已有部分，需补齐。

5. **handleSync 方法增加策略参数**：
```tsx
const handleSync = async (packageName: string, mode: 'overwrite' | 'skip') => {
  setSyncingPackage(packageName);
  try {
    const pkg = packages.find(p => p.name === packageName);
    const confirm = {
      mode,
      workflowAction: mode,
      roleActions: [], // 空数组表示全部使用 mode
      assetActions: [], // 空数组表示全部使用 mode
    };
    await api.teamPackages.syncPackage(packageName, confirm, pkg?.marketId);
    message.success(`团队包 ${packageName} 导入成功`);
    loadPackages();
    setPreviewModalVisible(false);
  } catch (error: any) {
    message.error(error.response?.data?.error || '导入失败');
  } finally {
    setSyncingPackage(null);
  }
};
```

#### 2.3 批量导入改造

**确认弹框改造**：

1. **调用预览 API 获取每个包的冲突信息**：
```tsx
const handleBatchImportClick = async () => {
  if (selectedRowKeys.length === 0) {
    message.warning('请先选择要导入的团队包');
    return;
  }

  const toImport = packages.filter(pkg =>
    selectedRowKeys.includes(`${pkg.marketId}-${pkg.name}`)
  );

  // 预览每个包获取冲突信息
  setLoadingPreview(true);
  const previews: PackagePreviewResponse[] = [];
  for (const pkg of toImport) {
    try {
      const result = await api.teamPackages.previewPackage(pkg.name, pkg.marketId);
      previews.push(result);
    } catch (error) {
      // 单个预览失败不阻塞整体流程
    }
  }
  setLoadingPreview(false);

  setBatchPreviews(previews);
  setPendingImportPackages(toImport);
  setConfirmModalVisible(true);
};
```

## 风险与约束

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 后端 repo 注入不完整 | 冲突检测不准确 | 开发前确认所需 repo 列表 |
| 前端样式与深色模式不兼容 | UI 问题 | 使用 CSS 变量 |
| 批量预览耗时较长 | 用户体验 | 可考虑并行请求或后端批量预览 API |

## Why

用户要求优化市场管理下的团队包导入能力，参考管理工具的实现，增加冲突检测和策略选择功能。
