## 一、协作守则

### 1.1 A2A 出口检查（必须执行）

**触发时机**: Agent 回复前必问

**检查问题**: "到我这里结束了吗？"

**三问短路规则**:
1. 需要对方采取行动？= 是 → 直接 @（跳过后续问题）
2. 对方需要知道这个信息？= 否 → 不 @
3. 会影响对方的工作？= 否 → 不 @

**结果**:
- 三个都否 → 不 @，任务结束
- 任一是 → @ 下游 Agent

### 1.2 @mention 格式规则（必须遵守）

**格式**: `@句柄` 必须单独成行，不能嵌入句子中

```
✅ 正确示例：
@后端开发工程师 请实现用户登录 API

❌ 错误示例：
确认后我将 @后端开发工程师 进行实现  ← 无效，不会触发
```

**原因**: 行首检测机制，嵌入句子中的 @ 会被忽略

### 1.3 角色边界（必须遵守）

**原则**: 各司其职，不越界

- **需求分析师**: 输出需求文档，不写代码
- **架构师**: 输出设计文档，不写代码
- **开发工程师**: 实现代码，不做架构决策
- **测试工程师**: 编写测试，不修改业务代码
- **代码审查工程师**: 审查代码，不直接修改

## 二、质量约束

### 2.1 根因优先（Bug 处理）

**原则**: 遇到 Bug 先定位根因，不要盲目试错

**流程**:
1. 复现问题 → 记录现象
2. 分析日志 → 定位位置
3. 检查代码 → 找到根因
4. 修复代码 → 解决问题

**禁止**: 猜测性修改、堆砌 try-catch、忽略错误

### 2.2 不确定先确认（决策）

**原则**: 对方向有疑问时先确认再继续

**触发条件**:
- 需求描述模糊
- 技术方案有歧义
- 影响范围不确定

**动作**: @需求分析师 或 @铲屎官 确认

### 2.3 Scope 守则（边界）

**原则**: 发现需求超出职责范围时，记录并停止

**触发条件**:
- 任务涉及其他角色职责
- 需求变更影响架构
- 发现潜在风险

**动作**: 
1. 记录问题
2. @ 触发者说明
3. 停止当前任务，等待决策

## 三、交接规范（五件套）

### 3.1 A2A Handoff 结构

**触发时机**: @ 下游 Agent 时，在回复开头输出

**格式**:
```
<a2a-handoff>
### What
具体改了什么，涉及哪些文件（包含文件路径和操作：Create/Edit/Delete）

### Why
为什么要这样做，约束/风险/目标是什么

### Tradeoff
放弃了什么备选方案

### Open Questions
还有什么不确定的

### Next Action
希望下游 Agent 做什么
</a2a-handoff>
```

**重要**:
1. 交接块必须放在回复开头，确保不被截断
2. What 部分必须列出涉及的文件路径和操作类型
3. 如果不需要 @ 下游 Agent，可以不输出交接块

### 3.2 Token 预算控制

**原则**: 交接块总 Token ≤ 800

**截断策略**: 首尾截断（Head 40%, Tail 60%）

## 四、Token 预算管理

### 4.1 上下文窗口配置

| 模型 | 窗口大小 |
|------|----------|
| claude-opus-4-6 | 200K |
| claude-sonnet-4-6 | 200K |
| claude-haiku-4-5 | 200K |
| gpt-5.3 | 128K |
| o3 | 200K |

### 4.2 A2A 深度限制

**静态上限**: 15 层

**动态计算**: `remainingBudget / avgTurnTokens`

**逻辑**: Token 预算不足时，自动降低深度上限

```
## 协作守则
- 出口检查：回复前问"到我这里结束了吗？"→ 三问短路 → @ 或不

- Scope 越界时记录并停止

## 交接规范

```
