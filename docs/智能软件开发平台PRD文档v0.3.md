## 1. 产品概述

### 1.1 需求背景

AI辅助研发正在深刻改变软件开发模式。传统开发流程中，从想法到产品需要经历需求分析、架构设计、编码实现、测试验证、部署上线等漫长周期。随着AI编程能力的提升，开发者可以跳过大量重复性编码工作，专注于创意和业务逻辑。

本平台旨在构建一个**项目级多Agent协同开发系统**，将用户的想法快速转换为可运行的项目，让用户无需关注开发过程、环境配置、部署流程等技术细节。

### 1.2 核心目标

**一句话愿景**：让有想法的人都能快速构建自己的产品。

核心目标：
- 降低产品开发门槛，让非技术用户也能实现创意
- 提升开发效率，让用户从重复工作中解放
- 提供全流程自动化，从需求到部署一站式完成
- 通过多Agent、多角色协同提升个人级项目开发效率

### 1.3 目标用户

| 用户类型   | 特征                 | 核心诉求                       | 成功标准              |
| ---------- | -------------------- | ------------------------------ | --------------------- |
| 非技术用户 | 有想法但缺乏编程能力 | 简单输入想法，获得可运行产品   | 5分钟内获得可预览产品 |
| 专业开发者 | 有技术背景，追求效率 | 自动化重复工作，专注于核心逻辑 | 减少70%重复编码时间   |

### 1.4 产品定位与差异化优势

#### 产品定位
面向个人开发者和非技术用户的智能化软件开发平台，通过AI驱动的多Agent协同系统，实现从创意到产品的全流程自动化。

#### 差异化优势
| 维度 | 传统开发工具 | 本平台 |
|------|-------------|--------|
| 使用门槛 | 需要编程知识 | 自然语言输入即可 |
| 开发周期 | 周级/月级 | 分钟级/小时级 |
| 环境配置 | 手动配置 | 自动化沙箱 |
| 流程管理 | 人工协调 | Agent协同 |
| 适用人群 | 技术人员 | 所有人 |

#### 2.1.3 开发工作台（对话交互界面）

**需求来源**：优化用户交互体验，实现多Agent协同的统一对话框交互模式。

**设计决策**：

| 决策项 | 选择 | 说明 |
|--------|------|------|
| Agent响应模式 | 顺序接力 | Agent按工作流顺序依次响应，前一个完成后下一个开始 |
| 用户控制权 | 混合模式 | 默认自动执行，用户可随时干预，干预后可恢复自动流程 |
| 可视化方式 | 统一消息流 | 所有Agent的消息在同一对话区展示，类似群聊 |
| 界面方案 | 增强式聊天 | 统一消息流 + 顶部可折叠进度条 |

**设计理由**：
1. 保持统一消息流的简洁感
2. 顶部进度条提供必要的上下文信息
3. 符合混合控制模式的需求（可随时干预、可跳转）
4. 与Trae等主流AI开发工具的交互模式一致，用户熟悉

**界面组成**：
```
┌─────────────────────────────────────────────────────────────────┐
│ 顶部进度条（可折叠）                                              │
│ [需求分析 ✓] → [方案设计 ●] → [代码实现] → [审查] → [部署]      │
│ 当前: 架构师 正在生成技术方案...                    [⏸ 暂停]     │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│                        对话消息区                                 │
│                    （统一消息流，可滚动）                         │
│                                                                  │
│ 🧑 用户: 帮我开发一个博客系统                                    │
│ 👤 需求分析师: 收到！我来分析你的需求...                         │
│ 🏗️ 架构师: 基于需求，我设计了以下技术方案...                     │
│ 💻 开发者: 正在编写代码...                                       │
│                                                                  │
├─────────────────────────────────────────────────────────────────┤
│ [📎] [输入消息或指令...]                        [发送] [干预]    │
│ [⏸ 暂停] [⏭ 跳过] [🔄 重做] [📋 产物]                           │
└─────────────────────────────────────────────────────────────────┘
```

| 区域 | 功能 | 说明 |
|------|------|------|
| 顶部进度条 | 显示开发进度、当前阶段、活跃Agent | 可折叠，默认展开 |
| 对话消息区 | 显示用户和Agent的对话消息 | 统一消息流，支持滚动 |
| 侧边快捷面板 | 快速访问产物、文件列表 | 可选展开，默认收起 |
| 底部控制栏 | 输入消息、快捷操作按钮 | 始终可见 |

**交互模式**：
- **默认自动执行**：Agent按工作流自动接力执行
- **用户可干预**：随时暂停、修改指令、跳转阶段
- **检查点确认**：关键节点需用户确认

**干预操作**：
| 操作 | 说明 |
|------|------|
| 与当前Agent对话 | 暂停自动流程，直接向Agent发送指令 |
| 跳转到指定Agent | 手动触发指定Agent执行 |
| 修改当前产物 | 编辑需求文档/架构图/代码 |
| 调整工作流 | 跳过/重做某阶段 |

**消息类型**：
| 类型 | 发送者 | 说明 |
|------|--------|------|
| 用户消息 | 用户 | 用户输入的文本或指令 |
| Agent消息 | Agent | Agent的回复内容 |
| 系统消息 | 系统 | 阶段切换、状态变更通知 |
| 产物卡片 | Agent | 生成的文档、代码、架构图等 |

**阶段状态**：
| 状态 | 标识 | 说明 |
|------|------|------|
| 已完成 | ✓ | 该阶段已完成 |
| 运行中 | ● | 当前正在执行 |
| 等待 | - | 等待前置阶段完成 |
| 需确认 | ⚠️ | 需要用户确认 |

### 2.2 多Agent协同系统

#### 2.2.1 Agent角色设计

**Agent角色定义**：

| Agent角色 | 职责 | 核心能力 |
|-----------|------|----------|
| 需求分析师 | 需求理解与结构化 | NLP解析、需求建模 |
| 架构师 | 方案设计与决策 | 架构设计、技术选型 |
| 开发者 | 代码实现 | 编码、调试 |
| 审查员 | 代码审查 | 静态分析、安全检测 |
| 测试工程师 | 测试验证 | 测试设计、测试执行 |
| 运维工程师 | 部署发布 | 环境配置、服务部署 |


**Agent角色描述规范**（用于路由匹配）：

```yaml
Agent角色描述模板:
  id: 需求分析师

  # 路由信号（系统读取）
  routing:
    use_when:
      - 用户输入自然语言需求
      - 需要解析和结构化需求
      - 需求变更需要更新文档
    not_for:
      - 技术方案设计（找架构师）
      - 代码编写（找开发者）
      - 代码审查（找审查员）
    output:
      - 需求文档（Markdown）
      - 功能列表（结构化）
      - 验收标准（AC清单）

  # 可读描述（人看的）
  description: |
    负责理解和结构化用户需求。
    当用户描述想法时，我会将其转化为
    清晰的需求文档和验收标准。
```

**阶段与Agent对应关系**：

| 阶段 | 主导Agent | 协作Agent | 主要产物 |
|------|-----------|-----------|----------|
| 需求分析 | 需求分析师 | - | 需求文档 |
| 方案设计 | 架构师 | 需求分析师（澄清） | 架构图、API文档、技术选型报告 |
| 代码实现 | 开发者 | 架构师（指导） | 源代码、配置文件 |
| 代码审查 | 审查员 | 开发者（修复） | 审查报告 |
| 测试验证 | 测试工程师 | 开发者（修复） | 测试报告 |
| 沙箱部署 | 运维工程师 | - | 部署产物、运行日志 |

**阶段流转规则**：
```yaml
PhaseFlow:
  需求分析:
    next: 方案设计
    checkpoint: 需求确认
    on_reject: 返回需求分析

  方案设计:
    next: 代码实现
    checkpoint: 方案确认
    on_reject: 返回方案设计 或 需求分析

  代码实现:
    next: 代码审查
    checkpoint: 无
    on_reject: 返回代码实现

  代码审查:
    next: 测试验证
    checkpoint: 代码合入
    on_reject: 返回代码实现
    on_p1_p2: 开发者修复

  测试验证:
    next: 沙箱部署
    checkpoint: 部署确认
    on_reject: 返回代码实现

  沙箱部署:
    next: 完成
    checkpoint: 无
    on_fail: 重试 或 返回
```

#### 2.2.2 Agent启动及运行
**功能描述**：让Agent能在一个聊天框中协同

必须：通过spawn CLI 作为子进程去拉起Agent，不能直接调用 Chat API或使用 Agent SDK，每个Agent是一个独立的实例，要支持Agent类型扩展
```
import { spawnCli } from '../../../utils/cli-spawn.js';

/** CLI flags for security boundary (statically scannable) */
const ALLOWED_TOOLS = 'Read,Edit,Glob,Grep';
const PERMISSION_MODE = 'acceptEdits';

export class ClaudeAgentService implements AgentService {
async *invoke(prompt: string, options?: AgentServiceOptions) {
const args: string[] = [
'-p', prompt,
'--output-format', 'stream-json',
'--verbose',
'--model', this.model,
'--allowedTools', ALLOWED_TOOLS,
'--permission-mode', PERMISSION_MODE,
];

    if (options?.sessionId) {
      args.push('--resume', options.sessionId);
    }

    const events = spawnCli({ command: 'claude', args });

    for await (const event of events) {
      const result = transformClaudeEvent(event, CAT_ID);
      if (result) yield result;
    }

    yield { type: 'done', catId: CAT_ID, timestamp: Date.now() };
}
}
```
**编排模式**：
```yaml
WorkflowStrategies:
  sequential:           # 顺序执行（默认）
    description: 按流程顺序执行，适合有依赖关系的任务
    pattern: "需求→设计→开发→审查→测试→部署"
    user_checkpoints:   # 用户确认点
      - 需求确认
      - 方案确认
      - 部署确认
```

### 2.3 Agent协同机制设计 ★新增

#### 2.3.1 A2A路由机制

**功能描述**：支持Agent之间通过@mention互相触发协作

**核心设计**：
- **触发方式**：行首`@Agent名`（防止代码注释、文档中的误触发）
- **执行模式**：Worklist串行执行
- **深度限制**：≤15轮（防止无限递归）
- **取消机制**：共享AbortController，一键终止全链
- **多目标限制**：单次最多@2个Agent（防止扇形爆炸）

**路由流程**：
```
用户/Agent消息
    ↓
@mention解析器（行首匹配）
    ↓
Worklist队列（动态追加）
    ↓
串行执行引擎（共享AbortController）
    ↓
直到Worklist清空
```

**Worklist执行伪代码**：
```typescript
async function* routeSerial(worklist: AgentId[], signal: AbortSignal) {
  let a2aCount = 0;
  const maxDepth = 15;

  for (let i = 0; i < worklist.length && a2aCount < maxDepth; i++) {
    if (signal.aborted) break;  // 用户取消

    const agentId = worklist[i];

    // 执行Agent
    for await (const msg of invokeAgent(agentId)) {
      yield msg;
    }

    // 检测@mention，追加到worklist
    const mentions = parseMentions(lastMessage, agentId);
    if (mentions.length > 0) {
      worklist.push(...mentions.slice(0, 2));  // 最多2个
      a2aCount++;
    }
  }
}
```

#### 2.3.2 MCP回传机制

**功能描述**：让Agent在执行过程中主动发言，而非等任务结束

**核心理念**：
- CLI输出 = Agent内心独白（私有）
- post_message = Agent主动发言（公开）

**核心能力**：
| 工具 | 功能 | 使用场景 |
|------|------|----------|
| post_message | 发送消息到项目聊天室 | 汇报进度、@其他Agent |
| thread_context | 获取对话上下文 | 理解历史讨论 |
| update_task | 更新任务状态 | 标记完成、失败 |
| request_permission | 请求危险操作授权 | 删除文件、执行敏感命令 |

**认证方式**：
- 双UUID认证（invocationId + callbackToken）
- Token有TTL，调用结束后失效
- 通过环境变量传递给Agent进程

**隐私模式配置**：
```yaml
PrivacySettings:

    - agent_id: 开发者

  # 适用场景
  use_cases:
    - 场景: 正常开发
      推荐设置: 关闭（减少信息噪音）
    - 场景: 调试/学习
      推荐设置: 开启（观察Agent思考过程）
```

### 2.4 上下文工程体系 ★新增

#### 2.4.1 设计原则

> **上下文工程 ≠ 给Agent更多信息**
> **上下文工程 = 让正确的信息，在正确的时间，出现在正确的角色面前**

Token数量不重要。549行的上下文塞满噪声，不如109行的精准版本。

### 2.5 协作元规则体系 ★新增

#### 2.5.1 设计原则

从AI弱点出发设计规则，而非照搬"别人怎么做"。

| AI弱点 | 会导致的问题 | 规则 |
|--------|-------------|------|
| 幻觉 | 编造不存在的信息 | 不确定就提问，不要硬猜 |
| 讨好 | Review只说"looks good" | Review必须铁面无私 |
| 缺乏记忆 | 交接信息丢失 | 交接必须写WHY |
| 过度自信 | 自己判断就合入 | 合入前必须经确认 |

#### 2.5.2 交接五件套

Agent间交接必须包含：

| 项目 | 说明 | 重要性 |
|------|------|--------|
| **What** | 具体做了什么 | 必需 |
| **Why** | 为什么这样做 | **最重要** |
| **Tradeoff** | 放弃了什么方案 | 建议 |
| **Open Questions** | 不确定的点 | 建议 |
| **Next Action** | 希望对方做什么 | 必需 |

**交接模板**：
```markdown
## 交接报告

### What - 我做了什么
修改了3个文件：
- user_service.go: 新增用户注册逻辑
- auth_middleware.go: 添加JWT验证
- user_test.go: 新增15个单元测试

### Why - 为什么这样做
用户注册需要验证邮箱，所以添加了邮件发送逻辑。
选择JWT而不是Session，因为需要支持移动端。

### Tradeoff - 放弃了什么方案
曾考虑用Redis存储Session，但为了简化部署，选了JWT。
曾考虑用第三方邮件服务，但用户要求自建，所以用SMTP。

### Open Questions - 不确定的点
1. 邮件模板放在哪个目录？暂时放在templates/
2. 密码强度规则用户没说，暂时用了默认规则

### Next Action - 希望对方做什么
@审查员 请帮我review代码质量和安全性
```

#### 2.5.3 Review分级标准

| 级别 | 定义 | 处理方式 |
|------|------|----------|
| **P1** | 阻断级：功能错误/安全问题 | **必须立即修** |
| **P2** | 重要级：质量/测试问题 | **必须修完才能放行** |
| **P3** | 建议级：风格/命名 | 可登记Backlog |

**Review输出模板**：
```markdown
## Review报告 #R1

### P1（共2个）— 必须立即修
1. user_service.go:132 - SQL拼接存在注入风险
   建议：使用参数化查询

2. auth_middleware.go:45 - 未处理token过期
   建议：添加过期检查和刷新逻辑

### P2（共1个）— 必须修完才能放行
1. user_test.go - 缺少边界情况测试
   建议：添加空输入、超长输入测试

### P3（共1个）— 可登记Backlog
1. 变量命名：user_service.go:20 - 变量tmp命名不清晰

### 结论
2 P1 + 1 P2 未清零，**不能放行**
```

**放行信号**：
```
✅ 可以放行：
  - "P1/P2 全部清零，可以放行"
  - "LGTM，没有阻断问题"
  - "通过，代码质量达标"

❌ 不能放行：
  - "整体OK，但还有几个P1"
  - "只剩小问题了"
  - 没有明确的放行语句
```

#### 2.5.4 禁止表演性同意

收到Review反馈时：

**✅ 应该**：
- 直接开始修复（行动 > 言语）
- 复述技术问题（证明你理解了）
- 问澄清问题（如果不理解）
- Push back（如果Reviewer错了，用技术论证）

**❌ 禁止**：
- "You're absolutely right!"
- "Great point!"
- "Excellent feedback!"
- "Thanks for catching that!"
- "让我现在就改"（在理解问题之前）

**修复确认模板**：
```markdown
## 修复确认 - R1反馈

### P1-1：SQL注入风险
- **你的发现**：user_service.go:132 使用字符串拼接SQL
- **我的修复**：改用参数化查询，使用$1, $2占位符
- **验证**：新增测试用例验证注入攻击被阻止

### P1-2：Token过期处理
- **你的发现**：auth_middleware.go:45 未检查token过期
- **我的修复**：添加exp字段检查，过期返回401
- **验证**：新增测试用例验证过期token被正确拒绝

全部修复完成，请@审查员 确认
```

#### 2.5.5 合入门禁

代码合入前必须满足：

| 检查项 | 要求 |
|--------|------|
| P1/P2清零 | 必须全部修复 |
| 审查员放行 | 必须明确说"可以放行"或"LGTM" |
| 测试通过 | 必须全部通过 |
| 用户确认 | 如需要 |

### 5.3 多Agent安全边界 ★新增

#### 5.3.1 安全边界五原则

| 原则 | 描述 | 防止的问题 |
|------|------|-----------|
| **路径唯一性** | 同一语义操作只有一条执行路径 | 双重开火 |
| **深度上限** | 所有链式执行有深度限制（≤15） | 无限递归 |
| **取消传播** | 所有执行必须注册到取消机制 | 不可取消 |
| **Thread隔离** | 有状态对象显式绑定threadId | 跨thread污染 |
| **错误累积上限** | 关键节点必须有人工确认 | 0.95^n问题 |

#### 5.3.2 深度限制配置

```yaml
depth_limits:
  max_a2a_depth: 15      # Agent互调最大深度
  max_auto_fix_rounds: 5 # 自动修复最大轮数
  max_retry_count: 3     # 最大重试次数
```

#### 5.3.3 取消机制设计

所有Agent调用必须：
1. 注册到InvocationTracker
2. 绑定AbortController
3. 响应取消信号并清理资源

**取消流程**：
```
用户点Stop
    ↓
signal.abort()
    ↓
InvocationTracker.cancel(threadId)
    ↓
遍历所有注册的调用，发送终止信号
    ↓
等待进程退出或超时杀死
```

#### 5.3.4 Thread隔离规则

任何有状态对象必须显式绑定threadId：
- Session记录
- 任务队列
- 缓存条目
- Socket事件
- 定时器

**Guard模式**：
```typescript
if (event.threadId !== currentThreadId) return;
// 处理事件...
```

#### 5.3.5 人工检查点

| 检查点 | 触发时机 | 必须确认 |
|--------|----------|----------|
| 需求确认 | 需求分析完成后 | 用户 |
| 方案确认 | 架构设计完成后 | 用户 |
| 代码合入 | Review通过后 | 审查员 |
| 部署上线 | 测试通过后 | 用户 |

## 9. 风险与挑战

### 9.1 技术风险

| 风险 | 等级 | 应对策略 |
|------|------|----------|
| AI生成代码质量不稳定 | 高 | 多轮审查+测试验证+人工审核 |
| 沙箱环境资源消耗大 | 中 | 资源限制+容器复用+定时清理 |
| Agent协同复杂度高 | 中 | 清晰的角色边界+完善的工作流+深度限制 |
| 大型项目处理困难 | 中 | 项目拆分+增量生成 |
| 无限递归风险 | 高 | 深度限制≤15+取消传播+人工检查点 |

### 9.2 产品风险

| 风险 | 等级 | 应对策略 |
|------|------|----------|
| 用户期望过高 | 高 | 明确产品边界+渐进式体验 |
| 学习成本 | 中 | 引导教程+预设模板 |
| 信任问题 | 中 | 透明展示过程+可干预机制 |
| 上下文漂移 | 高 | 愿景守护机制+冷启动验证 |

### 9.3 安全风险

| 风险 | 等级 | 应对策略 |
|------|------|----------|
| 代码注入攻击 | 高 | 输入过滤+沙箱隔离+权限限制 |
| API密钥泄露 | 高 | 密钥加密+访问控制+审计日志 |
| 敏感数据暴露 | 中 | 数据脱敏+访问控制 |
| 跨thread数据泄露 | 中 | Thread隔离+显式绑定threadId |

### 9.4 已知问题与应对 ★新增

#### 9.4.1 运行时问题

| 问题 | 症状 | 根因 | 应对 |
|------|------|------|------|
| 双重开火 | 任务执行两次 | 多路径触发 | 路径统一 |
| 无限递归 | Agent互@不停 | 缺深度限制 | 上限15轮 |
| 不可取消 | Stop无效 | 未注册tracker | 强制注册 |
| 跨thread污染 | 数据串线 | 缺threadId绑定 | 显式绑定 |

#### 9.4.2 交付质量问题

| 问题 | 症状 | 根因 | 应对 |
|------|------|------|------|
| 上下文漂移 | 不符合需求 | AC有损压缩 | 愿景守护 |
| 测试剧场 | 测试过但有问题 | Agent自测自 | 冷启动验证 |
| 链条断裂 | Agent该@不@ | 抑制规则过多 | 平衡正反规则 |
| 表演性同意 | 答应了但没理解 | 讨好倾向 | 禁止客套话 |

#### 9.4.3 事故案例参考

**案例：无限乒乓事故**
- 时间：2026-02-14
- 现象：两个Agent同时输出，Stop无效
- 根因：双重开火 + 无限递归 + 不可取消叠加
- 教训：路径必须统一，深度必须限制，取消必须传播
