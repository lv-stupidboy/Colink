# 团队包批量导入克隆缓存优化需求分析

**日期**: 2026-04-24
**分析师**: Colink需求分析师

---

## 需求确认

### 问题场景
用户希望优化团队包批量导入性能，避免对同一个 git 地址重复克隆。

### 确认的问题范围

| 场景 | 当前行为 | 期望行为 |
|------|----------|----------|
| 同一市场批量导入多个包 | 市场仓库被克隆 N 次 | 市场仓库只克隆 1 次 |
| 同一包仓库多个包（不同 Source） | 包仓库被克隆 N 次 | 包仓库只克隆 1 次 |

### 用户选择

1. **优化范围**: 两者都要优化（市场仓库 + 包仓库）
2. **缓存策略**: 临时缓存（请求级）
   - 在单次批量请求期间缓存克隆结果
   - 请求结束后自动清理
   - 简单实现，无持久化开销

## 当前代码分析

### 批量导入流程

| 层级 | 组件 | 行为 | 问题 |
|------|------|------|------|
| 前端 | `handleBatchImportConfirm` | 逐项串行调用 `syncPackage` | 无批量API调用 |
| 后端 | `PreviewPackagesBatch` | 并行调用 `PreviewPackage`（并发5） | 每个包独立克隆 |
| 后端 | `SyncPackagesBatch` | 并行调用 `SyncPackage`（并发3） | 每个包独立克隆 |
| 服务 | `SyncPackage` | 克隆市场仓库 → 克隆包仓库 | 无缓存机制 |
| GitClient | `CloneFromURL` | 每次创建新临时目录 | 重复克隆相同URL |

### 重复克隆示例

**场景一：同一市场导入 5 个包**
```
包A (市场1) → 克隆市场1仓库 + 克隆包A仓库
包B (市场1) → 克隆市场1仓库 + 克隆包B仓库  ← 市场1重复克隆
包C (市场1) → 克隆市场1仓库 + 克隆包C仓库  ← 市场1重复克隆
包D (市场1) → 克隆市场1仓库 + 克隆包D仓库  ← 市场1重复克隆
包E (市场1) → 克隆市场1仓库 + 克隆包E仓库  ← 市场1重复克隆
```

**场景二：同一包仓库 3 个包（不同 Source 子目录）**
```
包A (仓库X/SourceA) → 克隆仓库X
包B (仓库X/SourceB) → 克隆仓库X  ← 仓库X重复克隆
包C (仓库X/SourceC) → 克隆仓库X  ← 仓库X重复克隆
```

## 设计方案

### 架构

```
批量请求入口
      ↓
创建 CloneCache（请求级）
      ↓
并行处理 N 个包（使用 CloneWithCache）
      │
      ├─→ 包A：CloneWithCache(市场1) → 新克隆 → 存入缓存
      ├─→ 包B：CloneWithCache(市场1) → 从缓存返回 ← 不重复克隆
      ├─→ 包C：CloneWithCache(市场1) → 从缓存返回 ← 不重复克隆
      └─→ ...
      ↓
请求结束 → cleanupCache（清理所有克隆目录）
```

### 关键文件

| 文件 | 变更类型 | 说明 |
|------|----------|------|
| `clone_cache.go` | 新增 | CloneCache 结构（线程安全） |
| `git_client.go` | 修改 | 新增 CloneWithCache 方法 |
| `service.go` | 修改 | 新增 SyncPackageWithCache/PreviewPackageWithCache |
| `batch.go` | 修改 | 使用缓存，请求结束后清理 |

### API 变更

**无 API 变更**：前端无需修改，行为保持一致。
